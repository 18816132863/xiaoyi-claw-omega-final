# 小艺Claw新增技能接入规则

## 一、总原则

1. **主架构只保留稳定的六层**，不再因为新增功能继续加层
2. **新增技能优先作为"能力模块"挂入既有层**，不允许为单个功能单独造一层
3. **架构负责约束边界**，不负责直接拖着命令运行
4. **所有新增技能必须先注册、再接入、后放量**，未注册的能力一律不能进入主流程

---

## 二、固定主架构（6层）

| 层级 | 职责 | 允许放什么 | 禁止事项 |
|-----|------|----------|---------|
| **L1 表达层** | 对外展示、输入输出、UI/指令入口 | 前端页面、聊天入口、展示组件、表单 | 把业务规则写死在界面里 |
| **L2 应用编排层** | 流程编排、任务分发、会话控制、权限校验 | 工作流、任务路由、执行顺序、状态机 | 直接操作数据库或堆业务细节 |
| **L3 领域规则层** | 核心业务规则、判断逻辑、能力边界 | 规则引擎、业务判断、策略选择、领域对象 | 夹带网络调用或页面逻辑 |
| **L4 服务能力层** | 可复用技能、工具封装、模型调用、外部接口适配 | 新增技能模块、API封装、模型能力、工具能力 | 跨层乱读写或互相硬耦合 |
| **L5 数据访问层** | 数据读写、缓存、检索、索引、文件存取 | Repository、DAO、缓存服务、向量库适配 | 夹业务规则和流程编排 |
| **L6 基础设施层** | 日志、监控、消息、配置、部署、鉴权底座 | 中间件、消息队列、配置中心、观测系统 | 承载具体业务判断 |

---

## 三、新增技能先归哪一层

| 新增内容类型 | 首选层 | 说明 |
|------------|-------|------|
| 纯业务判断/新规则 | L3 领域规则层 | 例如新增判定标准、过滤逻辑、策略切换 |
| 可复用工具/新技能接口 | L4 服务能力层 | 例如OCR、检索、总结、翻译、外部系统调用 |
| 流程串联/自动执行链路 | L2 应用编排层 | 例如多技能顺序执行、回退、重试、审批 |
| 新数据源/新库存取 | L5 数据访问层 | 例如新增表、缓存、文件、向量库 |
| 日志监控/权限鉴权/配置 | L6 基础设施层 | 只做底座能力，不承载业务含义 |
| 新页面/新交互入口 | L1 表达层 | 只负责展示与交互，不写业务决策 |

---

## 四、接入标准七步（必须完整执行）

### 步骤1：能力归类
先判断新技能属于规则、服务、编排、数据还是基础设施；必须明确归属层。

### 步骤2：定义边界
写清输入、输出、依赖、失败条件、超时、回退方式；不允许模糊接入。

### 步骤3：标准接口
统一接口名、参数名、返回结构、错误码，禁止每个技能自己发明一套。

### 步骤4：注册入表
在统一注册表写入元数据；未入表视为不可用能力。

**注册表路径**：`infrastructure/inventory/skill_registry.json`

### 步骤5：灰度接入
先在测试链路或灰度流量里接入，验证兼容性、耗时、异常率。

### 步骤6：监控校验
补齐日志、调用链、耗时、成功率、版本号、负责人。

### 步骤7：正式放量
通过验收后再并入主流程；若不达标，回滚到前一版本。

---

## 五、统一技能注册表字段（必须有）

| 字段 | 是否必填 | 示例 | 说明 |
|-----|---------|------|------|
| skill_id | 是 | skill_ocr_v1 | 唯一标识，不能重名 |
| skill_name | 是 | OCR识别 | 面向人可读名称 |
| version | 是 | 1.0.0 | 版本号，便于回滚 |
| entry_layer | 是 | L4 | 明确挂在哪一层 |
| input_schema | 是 | file_url/lang | 输入参数定义 |
| output_schema | 是 | text/confidence | 返回结构定义 |
| dependencies | 是 | storage,model_api | 依赖的服务或库 |
| owner | 是 | 模块负责人 | 谁维护、谁兜底 |
| timeout_ms | 是 | 15000 | 超时阈值 |
| fallback_strategy | 是 | retry->degrade | 失败后的兜底策略 |
| status | 是 | draft/gray/prod/offline | 生命周期状态 |
| call_scope | 否 | internal only | 谁可以调用 |

---

## 六、约束规则（必须执行）

1. **禁止跨层直接调用**：上层只能调下一层公开接口，不能绕过应用层直插数据库
2. **禁止一个技能同时承担两种职责**：例如既做规则判断又做数据存取，这种必须拆开
3. **禁止主流程里塞临时补丁**：临时逻辑超过一次复用，就要沉淀成正式模块
4. **禁止未登记能力上线**：没有skill_id、版本、负责人、回退策略，不能并入正式环境
5. **禁止接口漂移**：任何入参与返回值变更，都要改版本，不允许静默兼容破坏旧流程
6. **每月做一次架构清理**：合并重复技能、下线废弃能力、梳理高耦合点

---

## 七、验收口径

- **看层数**：主架构是否稳定为6层，没有继续裂变
- **看归属**：每个新增技能是否都能明确指出entry_layer
- **看注册**：是否都进入统一注册表，字段是否完整
- **看边界**：是否存在跨层乱调、职责混装、接口漂移
- **看运行**：是否补齐监控、超时、错误码、回退策略

---

## 八、执行结论

**以后新增技能，不是给架构加层，而是按六层主架构归类入位；先注册、后接入、再放量；任何不符合边界和注册标准的能力，一律不得进入主流程。**
