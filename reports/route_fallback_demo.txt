================================================================================
路由 Fallback 演示
================================================================================

演示场景：各种失败情况下的 fallback 处理

--------------------------------------------------------------------------------
场景 1: 消息发送失败
--------------------------------------------------------------------------------

原始路由: route.send_message
失败原因: 网络错误 / 接收者离线 / 权限不足

Fallback 链:
1. query_message_status - 查询消息状态
2. list_recent_messages - 列出最近消息

执行流程:
┌─────────────────┐
│ send_message    │
│ (失败)          │
└────────┬────────┘
         ↓
┌─────────────────┐
│ query_message_  │
│ status          │
│ (获取失败原因)  │
└────────┬────────┘
         ↓
┌─────────────────┐
│ 向用户报告:     │
│ "消息发送失败， │
│ 原因: XXX"      │
└─────────────────┘

--------------------------------------------------------------------------------
场景 2: 删除备忘录失败
--------------------------------------------------------------------------------

原始路由: route.delete_note
失败原因: 备忘录不存在 / 权限不足

Fallback 链:
1. query_note - 查询备忘录

执行流程:
┌─────────────────┐
│ delete_note     │
│ (失败)          │
└────────┬────────┘
         ↓
┌─────────────────┐
│ query_note      │
│ (查询当前状态)  │
└────────┬────────┘
         ↓
┌─────────────────┐
│ 向用户报告:     │
│ "删除失败，     │
│ 当前备忘录: ..."│
└─────────────────┘

--------------------------------------------------------------------------------
场景 3: 任务调度失败
--------------------------------------------------------------------------------

原始路由: route.schedule_task
失败原因: 调度器不可用 / 时间冲突

Fallback 链:
1. query_note - 记录到备忘录

执行流程:
┌─────────────────┐
│ schedule_task   │
│ (失败)          │
└────────┬────────┘
         ↓
┌─────────────────┐
│ query_note      │
│ (创建备忘录记录)│
└────────┬────────┘
         ↓
┌─────────────────┐
│ 向用户报告:     │
│ "任务调度失败， │
│ 已记录到备忘录" │
└─────────────────┘

--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
场景 5: 多级 Fallback 链
--------------------------------------------------------------------------------

假设路由: route.high_risk_operation
Fallback 链: medium_risk → low_risk → safe_default

执行流程:
┌─────────────────┐
│ high_risk_      │
│ operation       │
│ (失败)          │
└────────┬────────┘
         ↓
┌─────────────────┐
│ medium_risk     │
│ (失败)          │
└────────┬────────┘
         ↓
┌─────────────────┐
│ low_risk        │
│ (成功)          │
└────────┬────────┘
         ↓
┌─────────────────┐
│ 返回结果        │
└─────────────────┘

--------------------------------------------------------------------------------
场景 6: Fallback 风险降级
--------------------------------------------------------------------------------

原则: Fallback 路由的风险等级 ≤ 原路由

示例:
- send_message (HIGH) → query_message_status (LOW) ✓
- delete_note (HIGH) → query_note (LOW) ✓
- update_note (MEDIUM) → query_note (LOW) ✓

错误示例 (不允许):
- query_note (LOW) → delete_note (HIGH) ✗

--------------------------------------------------------------------------------
场景 7: 循环 Fallback 检测
--------------------------------------------------------------------------------

系统自动检测并阻止循环 fallback:

route.a → route.b → route.c → route.a (循环!)

检测机制:
- 记录已访问的路由
- 如果遇到已访问的路由，终止 fallback
- 返回最后一个成功的路由结果

--------------------------------------------------------------------------------
场景 8: 终端 Fallback
--------------------------------------------------------------------------------

终端 Fallback: 没有 fallback 的路由，通常是安全的只读操作

示例终端路由:
- query_note (无 fallback)
- get_location (无 fallback)
- diagnostics (无 fallback)

当到达终端路由时:
- 如果成功，返回结果
- 如果失败，向用户报告错误
